# Git Workflow

> Read by: arms-devops-agent
> Triggered by: commits, branching, releases, PR generation

---

## Branch Strategy

```
main          — production-ready, protected. Direct pushes blocked.
develop       — integration branch, auto-deploys to staging
feature/*     — individual features, branched from develop
fix/*         — bug fixes, branched from develop (or main for hotfixes)
hotfix/*      — critical production fixes, branched from main
chore/*       — tooling, deps, config changes
```

**Rule:** `arms-devops-agent` never pushes directly to `main`. All production changes go through PR → review → merge.

## Conventional Commits

All commits generated by ARMS must follow this format:

```
<type>(<scope>): <description>

[optional body]
[optional footer]
```

### Types
| Type | When to Use |
|---|---|
| `feat` | New feature |
| `fix` | Bug fix |
| `chore` | Build, deps, tooling — no production code change |
| `refactor` | Code restructure — no behaviour change |
| `test` | Adding or updating tests |
| `docs` | Documentation only |
| `style` | Formatting, whitespace — no logic change |
| `perf` | Performance improvement |
| `ci` | CI/CD config changes |
| `revert` | Reverting a previous commit |

### Scope Examples
```
feat(auth): add OAuth login with Google
fix(db): correct RLS policy on profiles table
chore(deps): upgrade Next.js to 15.2.0
test(api): add unit tests for users endpoint
refactor(components): extract Button into shared library
```

### Rules
- Description is lowercase, no period at end
- Present tense: "add feature" not "added feature"
- Under 72 characters for the subject line
- Body explains *why*, not *what*

## Checkpoint Commits

`arms-devops-agent` creates checkpoint commits before major work:

```bash
git add .
git commit -m "chore: checkpoint before [Task Name]"
```

Checkpoints are created before:
- Database migrations
- Large refactors
- Multi-agent parallel execution batches
- Any destructive operation

## PR Template

When creating pull requests, use this structure:

```markdown
## Summary
<!-- One sentence: what does this PR do? -->

## Changes
<!-- Bullet list of key changes -->

## Testing
- [ ] Unit tests pass (`npm run test`)
- [ ] E2E tests pass (`npx playwright test`)
- [ ] Build passes (`npm run build`)
- [ ] Manually tested on mobile viewport

## Screenshots (if UI changes)
<!-- Before / After -->

## Notes
<!-- Anything reviewers should know: trade-offs, follow-ups, known issues -->
```

## Release Tagging

Tag every production release:

```bash
# Create annotated tag
git tag -a v1.2.0 -m "Release v1.2.0 — <brief description>"
git push origin v1.2.0
```

### Version Format (Semantic Versioning)
```
MAJOR.MINOR.PATCH

MAJOR — breaking changes
MINOR — new features, backward compatible
PATCH — bug fixes, backward compatible
```

### Auto-generate changelog from commits:
```bash
git log v1.1.0..HEAD --oneline --no-merges
```

## Hotfix Flow

For critical production bugs:

```bash
# Branch from main (not develop)
git checkout main
git pull origin main
git checkout -b hotfix/critical-auth-bug

# Fix, test, commit
git commit -m "fix(auth): patch session expiry race condition"

# Merge to both main AND develop
git checkout main && git merge hotfix/critical-auth-bug
git checkout develop && git merge hotfix/critical-auth-bug

# Tag the fix
git tag -a v1.1.1 -m "Hotfix: auth session expiry"
git push origin main develop --tags
```
